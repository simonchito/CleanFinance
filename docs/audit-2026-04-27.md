# Auditoría 2026-04-27

## Alcance

Revisión integral del repo Flutter CleanFinance: arquitectura, tests, documentación, ARB/i18n, seguridad, Android, dependencias, robustez funcional y calidad de código.

## Hallazgos por prioridad

### Crítico

- No se detectaron fallos críticos reproducibles: `flutter analyze` y `flutter test` pasaban antes de aplicar cambios.

### Alto

- Portugués figuraba como idioma soportado (`pt`) pero el ARB estaba incompleto y mezclaba textos en español. En el follow-up de esta auditoría se restauró soporte activo para portugués genérico (`pt`) y variantes como `pt_BR`/`pt_PT` se normalizan a `pt`.
- Logs técnicos de startup, SQLite, secure storage, auth y biometría podían imprimir rutas, errores o stack traces fuera de debug. Se restringieron a `kDebugMode`.

### Medio

- `.gitignore` ignoraba `android/`, lo que podía ocultar cambios relevantes de configuración Android. Se quitó ese ignore amplio y se agregaron ignores explícitos para secretos de firma.
- Documentación desactualizada: versión de DB, `compileSdk`, idiomas soportados y signing release no reflejaban completamente el código actual.

### Bajo

- Faltaba un test que protegiera la paridad de ARB y el set real de idiomas soportados.
- Dependencias con versiones nuevas disponibles: se documenta como evaluación separada porque la suite está verde y varias actualizaciones son saltos mayores.

## Follow-up 2026-04-27: segunda pasada focalizada

Corrección importante:

- CleanFinance mantiene 3 idiomas activos: Español (`es`), Inglés (`en`) y Portugués genérico (`pt`).
- La decisión de retirar portugués queda revertida. No se reintroduce `pt_BR` como idioma separado: preferencias y locales de dispositivo `pt_BR`, `pt_PT` y otras variantes portuguesas se normalizan de forma segura a `pt`.

### Cambios aplicados

- Restaurado `lib/l10n/app_pt.arb`, completando paridad con `app_es.arb` y `app_en.arb` y corrigiendo textos portugueses que habían quedado en español.
- Regeneradas localizaciones, incluyendo `AppLocalizationsPt`.
- Actualizados `AppLocaleMode`, `AppConstants`, `AppStrings`, `CleanFinanceApp` vía `supportedLocales`, y la selección de idioma en Ajustes para ofrecer Sistema, Español, Inglés y Portugués.
- Subida la versión de DB a 10 para normalizar variantes portuguesas legacy (`pt_*`) a `pt`; la migración de versión 8 también conserva `pt`.
- Privacidad de montos unificada en Dashboard/Home, Movements, Reports, Savings, Budgets, gráficos e insights mediante `AmountVisibilityFormatter` y `showSensitiveAmountsProvider`.
- Dependencias patch aplicadas: `sqflite` 2.4.2 -> 2.4.2+1, `sqflite_common` 2.5.6 -> 2.5.6+1, `mocktail` 1.0.4 -> 1.0.5.

### Tests actualizados

- `test/app_locale_mode_test.dart`: valida `pt`, `pt_BR`, `pt_PT` y otra variante portuguesa como portugués genérico, resolución de locales del sistema y fallback a español.
- `test/l10n_consistency_test.dart`: exige archivos ARB `es/en/pt`, paridad de claves y soporte generado para `pt`.

### Build Android

- `flutter build apk --debug` dentro del sandbox quedó sin salida hasta timeout de 5 minutos.
- Repetido fuera del sandbox para usar Gradle/Android SDK real: compiló correctamente en 51.5s y generó `build/app/outputs/flutter-apk/app-debug.apk`.
- No hay evidencia de error de proyecto; el bloqueo observado corresponde al entorno sandbox.

### SQLite sin cifrado

Estado actual:

- SQLite local sigue sin cifrado a nivel archivo.
- PIN, recuperación y estado de bloqueo viven hasheados o protegidos en secure storage.
- Backups pueden exportarse como JSON plano o cifrados con contraseña opcional.

Riesgo:

- Si un atacante obtiene acceso al archivo SQLite del dispositivo, los datos financieros locales pueden leerse.

Alternativas:

- SQLCipher/sqflite_sqlcipher: mejor protección en reposo, pero requiere migración de DB, manejo de claves, validación Android/iOS y estrategia de recuperación.
- Cifrado selectivo de campos sensibles: menor alcance, pero complica consultas/reportes y puede romper agregados.
- Mantener SQLite plano y reforzar controles alrededor: menor riesgo técnico inmediato, pero no soluciona cifrado en reposo.

Recomendación:

- No implementar cifrado completo en esta pasada. Planificar una tarea dedicada con spike técnico, migración reversible y pruebas de backup/importación.

### Web/Desktop/iOS

- Notificaciones ya tienen fallback controlado: solo Android se considera plataforma soportada y el resto devuelve estado `unsupported`.
- Biometría y secure storage tienen guardas defensivas por excepción y degradan a PIN/estado no disponible.
- Soporte real Web/Desktop/iOS sigue pendiente de validación runtime porque SQLite con `sqflite`, secure storage, biometría y flujos de archivo/share dependen de plugins nativos.
